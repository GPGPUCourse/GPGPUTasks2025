# Теоретическое задание: параллелизуемость/code divergence/memory coalesced access

**1)** Пусть на вход дан сигнал x[n], а на выход нужно дать два сигнала y1[n] и y2[n]:

```
 y1[n] = x[n - 1] + x[n] + x[n + 1]
 y2[n] = y2[n - 2] + y2[n - 1] + x[n]
```

Какой из двух сигналов будет проще и быстрее реализовать в модели массового параллелизма на GPU и почему?

**Ответ**

Проще будет реализовать первый, так как нам не нужно будет ожидать, пока другие потоки доделают свои задачи для `n-1` и `n-2` и предоставят нам результат (отсутсвие необходимости в синхронизации между потоками). Также, за счет доступа к значениям с разных потоков, скорее всего мы не будем грузить их повторно - будет использоваться кэширование с локальной памятью.

**2)** Предположим что размер warp/wavefront равен 32 и рабочая группа делится на warp/wavefront-ы таким образом что внутри warp/wavefront номер WorkItem по оси x меняется чаще всего, затем по оси y и затем по оси z.

Напоминание: инструкция исполняется (пусть и отмаскированно) в каждом потоке warp/wavefront если хотя бы один поток выполняет эту инструкцию неотмаскированно. Если не все потоки выполняют эту инструкцию неотмаскированно - происходит т.н. code divergence.

Пусть размер рабочей группы (32, 32, 1)

```
int idx = get_local_id(1) + get_local_size(1) * get_local_id(0);
if (idx % 32 < 16)
    foo();
else
    bar();
```

Произойдет ли code divergence? Почему?

**Ответ**

`int idx = get_local_id(1) + get_local_size(1) * get_local_id(0);`\
`get_local_id(1)` - меняется редко (y)\
`get_local_size(1)` - константа\
`get_local_id(0)` - меняется часто (x)\
`z` - константа

Так как рабочая группа состоит из 32 warp-ов, то в одной рабочей группе все значения будут с разными x, но одинаковыми y и z:
|x |y |z |  |
|--|--|--|--|
|0 |0 |0 |  |
|...|0 |0 |  |
|31|0 |0 |<- первая рабочая группа (32 значения)|
|0 |1 |0 |  |
|...|1 |0 |  |
|31 |1 |0 |<- вторая рабочая группа (32 значения)|
|...|...|0 |  |
|31|31|0 | <- последняя рабочая группа|

Тогда значения idx для одной рабочей группы:\
`idx = y + 32 * x`\
`idx = const + 32 * (0..31)`

Тогда значения `idx % 32`:\
`(const + 32 * (0..31)) % 32 = const`

Следовательно code divergence не произойдет, как так для каждого warp значение `idx % 32` будет одинаковым.


**3)** Как и в прошлом задании предположим что размер warp/wavefront равен 32 и рабочая группа делится на warp/wavefront-ы таким образом что внутри warp/wavefront номер WorkItem по оси x меняется чаще всего, затем по оси y и затем по оси z.

Пусть размер рабочей группы (32, 32, 1).
Пусть data - указатель на массив float-данных в глобальной видеопамяти идеально выравненный (выравнен по 128 байтам, т.е. data % 128 == 0). И пусть размер кеш линии - 128 байт.

**(a)**

```
data[get_local_id(0) + get_local_size(0) * get_local_id(1)] = 1.0f;
```

Будет ли данное обращение к памяти coalesced? Сколько кеш линий записей произойдет в одной рабочей группе?

**Ответ**

```
data[x + x_size * y] = 1.0f;
```

Будем идти по данным подряд => coalesced

Размер кэш-линии - 128 байт\
Размер warp - 32 потока\
Каждое записанное значение (float) - 4 байта\
В одной кэш-линии: 128 / 4 = 32 значения

Тогда при последовательной записи один warp запишет: 32 * 4 = 128 байт.
Так как указатель на массив идеально выравнен по 128 байтам, то каждый warp будет писать только в одну свою кэш линию.
Следовательно: 32 кэш линии будет записано одной рабочей группой (по количеству warp-ов в группе).

**(b)**

```
data[get_local_id(1) + get_local_size(1) * get_local_id(0)] = 1.0f;
```

Будет ли данное обращение к памяти coalesced? Сколько кеш линий записей произойдет в одной рабочей группе?

**Ответ**

```
data[y + y_size * x] = 1.0f;
```

Будем обращаться прыжками по данным (записываем каждое 32-е значение) => не coalesced

Размер кэш-линии - 128 байт\
Размер warp - 32 потока\
Каждое записанное значение (float) - 4 байта\
В одной кэш-линии: 128 / 4 = 32 значения

В данном случае, так как значения, которые пишет warp, лежат с шагом в 32, то, для записи каждого нового значения, потребуется "записать" еще 31 его соседа по кэш-линии. 
Следовательно, будет записано 1024 кэш-линии будет записано одной рабочей группой (количеству warp-ов в группе * количество записанных значений в одном warp-е).

**(c)**

```
data[1 + get_local_id(0) + get_local_size(0) * get_local_id(1)] = 1.0f;
```

Будет ли данное обращение к памяти coalesced? Сколько кеш линий записей произойдет в одной рабочей группе?

**Ответ**

```
data[1 + x + x_size * y] = 1.0f;
```

Будем идти по данным подряд => почти coalesced, не считая сдвига

Размер кэш-линии - 128 байт\
Размер warp - 32 потока\
Каждое записанное значение (float) - 4 байта\
В одной кэш-линии: 128 / 4 = 32 значения

Тогда при последовательной записи один warp запишет: 32 * 4 = 128 байт. Указатель на массив идеально выравнен по 128 байтам, но мы добавляем сдвиг на 1 байт в коде, то есть, каждый warp будет писать в свою кэш линию и в соседнюю.
Следовательно: 64 кэш линии будет записано одной рабочей группой (по количеству warp-ов в группе * 2).
